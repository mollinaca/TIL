## 6-02 業務プロセスとモデル化

**業務プロセス（ビジネスプロセス）**  
情報システムのあるべき姿を明確にするため、DFDやE-R図などを用いて対象業務のモデル化を行う。

**エンタープライズアーキテクチャ**  
組織全体の業務とシステムを統一的な手法でモデル化し、業務とシステムを同時に改善することを目的とした最適化手法。

**DFD**  
Data Flow Diagram  
データの流れを表した図  

**E-R図**  
Entity-Relationship Diagram  
データの構造を、実体（Entity)と実体間の関連（Relationship）という概念を用いて表したもの

**UML**  
Unified Modeling Language  
オブジェクト指向によるシステム開発で利用される表記法  

**クラス図**  
長方形で表し、上段からクラス名、属性名、操作名を記述する。  
クラスを図表記したもの。

**ユースケース図**  
利用者とシステムのやり取りを表した図

**シーケンス図**  
オブジェクト間のメッセージの流れを時系列に沿って表した図

## 6-05 コード設計と入力チェック

ここでいうコードは、プログラムのコードではなく、社員コードや顧客コードなどの、オブジェクトを識別するラベルという意味でのコード。

**コードの種類**  

* 順番コード
  データの発生順や受付順に、一連番号を付ける

* 区分コード  
  コード化するデータをある特性により、いくつかのグループにわけ、グループ別に一連番号を付ける

* けた別コード
  コードの桁に意味をもたせ、最上位桁から大分類、中分類、小分類というように階層的に番号をつける

* 表意コード  
  コード化される対象データの名称や略称などをコードに組み込む

**入力チェック方式**

∥ 名称 | 概要 |
| - | - |
| ニューメリックチェック | 数値かどうか |
| シーケンスチェック | データが昇順や降順など決められた順番に並んでいるかどうか |
| 重複チェック | 重複したデータが存在しないかどうか |
| フォーマットチェック | データが決められた形式にそっているかどうか |
| 論理チェック | データが論理的に矛盾しないかどうか |
| リミットチェック | データの値が一定の範囲に収まってるかどうか |
| 照合チェック | データがマスタファイルに存在するかどうか |
| チェックディジットチェック | チェックディジットを使って確認する |

## 6-06 モジュール分割

プログラムを機能ごとに、詳細設計で適切なモジュールに分割、階層構造にすること。
ソフトウェア詳細設計。  

**分割技法**

データの流れに着目：STS分割法、トランザクション分割、共通機能分割法
データ構造に着目：ジャクソン法、ワーニエ法

| 名称 | 概要 |
| - | - |
| STS分割法 | 入力データを処理する部分、入力データを出力データに変換する部分、出力データを処理する部分に分割する |
| トランザクション分割法 | トランザクションの種類ごとにモジュール分割する |
| 共通機能分割法 | 共通な機能を一つにまとめ、共通モジュールとして独立させる |
| ジャクソン法 | 入出力データの構造に着目して分割する。基本・連続・選択・繰り返しの4つの基本図式を使う |
| ワーニエ法 | 扱うデータの構造に着目して、入出力データの構造図を作成し、入力データの構造図をもとにプログラム構造図を作成する |

**モジュール強度**  

* 機能的強度：モジュール内のすべての機能が、単一機能を実行するために関連している
* 情報的強度：特定のデータ構造を扱う複数の機能が一つのモジュールにまとまっている
* 連絡的強度：複数の逐次的な機能を実行し、モジュール内の機能間でデータを受け渡したり、同じデータを参照したりする
* 手順的強度：複数の逐次的な機能が一つのモジュールにまとまっている
* 時間的強度：ある時点で一括して実行できる機能が一つのモジュールにまとまっている
* 論理的強度：関連した複数の機能を一つにまとめており、どの機能が実行されるかは呼び出されるときの引数によって決定される
* 暗号的強度：複数のまったく関係のない機能を実行し、モジュールにあの各機能の間に特別の関係が認められない

**モジュール結合度**  

* データ結合：データ項目だけをモジュール間の引数として渡す
* スタンプ結合：データ構造をモジュール間の引数として渡す
* 制御結合：制御パラメタを引数として渡し、モジュールの実行順序を制御する
* 外部結合：外部宣言したデータを関係するモジュールが参照する
* 共通結合：共通域に定義したデータを関係するモジュールが参照する
* 内容結合：ほかのモジュール内のデータを直接参照する

上にいくほど疎結合で独立

## 6-07 プログラミング

構造化プログラミング → 順次、選択、繰り返し

* 値渡し/値呼び出し：実引数から仮引数へ値のコピーを渡す。仮引数を変更しても、実引数へ影響なし
* 参照渡し/参照呼出し：実引数から仮引数へ値が格納されているアドレスを渡す。仮引数を変更すると、アドレス空間を通じて実引数も値が変わる

**デバッグツール**

| 名称 | 概要 |
| - | - |
| トレーサ | プログラムの命令の実行順序、実行結果などの履歴情報 | 
| スナップショットダンプ | プログラム実行中の、指定されたメモリ領域の内容を出力する |
| ファイルダンプ | 磁気テープファイルやディスクファイルなどの内容をダンプする |
| メモリダンプ | プログラム実行中にエラーが発生した場合に、メモリの内容をダンプする |

## 6-09 テスト手法

プログラムのテストの目的はバグを発見することであり、可能な限りエラーを見つけることができるようにテストデータを作成する必要がある。
テスト開始後からの累積バグ件数をグラフにすると、通常、信頼度成長曲線（ゴンぺルツ曲線）と呼ばれる曲線になる。

・システム要件定義 → システム的確性確認テスト（システムテスト）  
 ↓  
・システム方式設計（外部設計） → システム結合テスト  
 ↓    
・ソフトウェア要件定義（外部設計） → ソフトウェア的確性確認テスト  
 ↓  
・ソフトウェア方式設計（内部設計） → ソフトウェア結合テスト（結合テスト）  
 ↓    
・ソフトウェア詳細設計（プログラム設計） → ソフトウェアユニットテスト（単体テスト）  
 ↓    
・ソフトウェア構築（プログラミング）

**ユニットテスト**  
単体テスト、プログラムを機能単位に分割したモジュール単位で行うテスト  

* ホワイトボックステスト
  内部構造に着目して行う

* ブラックボックステスト
  外部使用に着目して行う

**結合テスト**  
モジュール間のインタフェイスを検証する

**トップダウンテスト**  
上位のモジュールから下位のモジュールへ順次結合して検証する。  
下位モジュールが完成していない場合、テスト対象のモジュールからの呼び出し目入れの条件に合わせてダミーの値を返す**スタブ**が必要になる。

**ボトムアップテスト**  
下位のモジュールから上位のモジュールへと順次結合して検証する。  
上位モジュールが完成していない場合、テスト対象モジュールに引数を渡して呼び出すダミーモジュールの**ドライバ**が必要。

**ソフトウェア適格性確認テスト**  
ソフトウェア要件定義で定義したソフトウェア適格性要件に従って行い、ソフトウェア要件通りに実現されているかどうか検証する。

**システム結合テスト**  
システム方式設計で定義したテスト仕様に従い、システムが要件を満たしているか確認する。

**システム適格性テスト**  
実際に実務で使うデータや、業務上例外として処理されるデータを使って行うテスト

## 6-10 レビュー

設計の各工程ごとに設計品質の評価を行い、各工程が終了したかどうかを判断するために実施する検討会。  

**ラウンドロビン**  
参加者全員が持ち回りでレビュー責任者を務めながらレビューを行う

**ウォークスルー**  
レビュー対象物の作成者が説明者になり、入力データ値を過程して、手続きをステップごとにシミュレーションしながらレビューを行う。  

**インスペクション**  
第3者が責任者となり検証する

## 6-11 ITサービスマネジメント

ITIL(Infomation Technology Infrastructure Library)：アイティル  
ITサービスマネジメントに関するベストプラクティス  

| プロセス | 概要 |
| - | - |
| インシデント管理 | インシデントを最小限に抑えて速やかに回復させる |
| 問題管理 | インシデントの根本原因を突き止めて恒久的な解決策を提供 |
| サービス資産管理及び構成管理 | ITサービス提供に必要なアイテム（CI)を常に正しく把握し、最新状態にする |
| 変更管理 | 変更に伴う影響の検証・評価を行い、許可または却下する |
| リリース管理及び展開管理 | 変更管理されたITサービスを本番環境に展開する |

* サービスデスク：利用者からの問い合わせに対する単一の回答窓口
* インシデント管理：サービスを迅速に復旧させる
* 問題管理：問題を抜本的に解決する

**SLA(Service Level Agreement)**   
サービスの品質に関する利用者と提供者間の合意。 
ITサービスの維持・工場を図るマネジメント活動をサービスレベルマネジメントという。

**ファシリティマネジメント**  
IT関連設備について、適切な使われ方をしているかの監視を行う。
